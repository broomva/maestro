Maestro architecture

A 24/7 runtime owns the workspace and runs the loops. A thin relay brokers connections. Clients are projections that subscribe. Nothing important lives in the client.

The earlier local-daemon shape (a maestrod on the user's machine, OS timers) is retired. The runtime is always-on and off the client; the scheduler and run queue live in its control-plane index. Aligned with ARCHITECTURE.md and AUTONOMY.md in build-docs/.

Clients · projections, no source-of-truth state

Web / PWA Vite + React SPA

The primary surface: check on your runs from anywhere. The PWA is the installable form of the same SPA. Tokens and components come from this design system.

Desktop Tauri 2 shell

The same SPA in a native shell. Adds tray presence for the orchestrator, native "Needs you" notifications, keychain for the runtime credential. A client, never a runtime.

Marketing Next.js on Vercel

Separate build target for the landing site only. The app never uses Next; SSR means nothing to a socket-fed projection.

Runtime · 24/7, self-host (the trust tier) or managed

The runtime TypeScript (Bun/Node) + Claude Agent SDK

Owns a workspace (FS + git + sh), runs the four nested loops, spawns agents into isolated run/<id> sandboxes, emits the append-only event log. All agent logic lives here and only here. The Anthropic key lives here too, never in the client, never proxied through the relay.

Loop 1 · agent per step
The Agent SDK loop in a worktree. The atomic unit; don't over-build it.
Loop 2 · verifier per attempt · the whole game
Writer never grades its own homework. Deterministic oracle first; LLM judge only for subjective work, never the sole gate. Produces the judge-verdict receipt.
Loop 3 · triggers + the gate per event
Heartbeat · cron · hook · goal, with idempotency keys. The governance gate maps to the verbs: approve, send back, hold, point/grant.
Loop 4 · hill-climb per N runs
Trace analysis rewrites the inner loops. Human-reviewed in v1; automate last, change rarely.
Guardrails
Three stop conditions (iteration cap, no-progress halt, budget), budget checked in-path before each call, kill switch, audit trail.
Scheduler + run queue
Routines, cron, Standing work. Timers and leases live in the control-plane index, not the OS.
sandbox interface · a worktree and a container are the same abstraction

Phase 1 · git worktrees now

Logical isolation on the runtime host: one worktree per run on a run/<id> branch. Honest limit: runs share the host shell — acceptable when the runtime is the user's own.

Phase 2 · containers / microVMs later

Physical isolation per run, behind the same sandbox interface. A swap, not a rewrite.
